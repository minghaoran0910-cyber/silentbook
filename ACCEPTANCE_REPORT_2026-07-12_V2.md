## 二、P0 Bug 详情

### 🔴 Bug 1：交易联动资产余额失败（服务器端）

**现象：**
- 通过 API 创建消费交易（¥100, account=招商银行），交易记录创建成功（id=301）
- 但 accounts 表中招商银行余额完全不变（5816.17 → 5816.17）
- 删除交易后余额仍不变

**根因：**
- **服务器代码未部署最新版本**
- 服务器版本：`7b68320`（fix: 前端healthcheck修复）
- 本地版本：`4d2c67f`（V2-039: 交易联动余额 + 持仓同步资产）
- 缺少关键提交：`4d2c67f` 包含交易联动余额的实现代码

**本地验证：** ✅ 本地 Docker 环境交易联动正常（5816.17 → 5716.17 → 回滚 5816.17）

**修复方案：**
```bash
# 在服务器上部署最新代码
ssh root@39.97.59.29
cd /opt/silentbook
git pull origin main
docker compose -f docker-compose.server.yml up -d --build backend
```

**影响：** 🔴 阻塞上线。核心记账功能不完整，用户消费后余额不更新。

### ⚠️ Bug 2：账户表重复数据

**现象：**
- 服务器 accounts 表有 8 条记录（id 1-8），实际是 4 个账户的重复副本
- id 1-4 和 id 5-8 分别是相同的账户（招商银行、支付宝、微信、现金）

**根因：** 数据迁移或初始化脚本重复执行

**影响：** 可能导致交易关联到错误的账户记录

---

### ⚠️ Bug 3：0元垃圾交易仍在数据库

**现象：**
- 交易列表中存在大量 ¥0 的非财务通知（快递、营销、揽收提醒）
- 示例：id=299 "运输通知" ¥0、id=298 "海尔冷柜" ¥0

**分析：**
- 通知解析器的白名单功能正常工作（返回 status=filtered）
- 但被过滤的通知仍以 ¥0 记录到数据库
- 这是设计问题：过滤 ≠ 不记录

**影响：** 交易笔数虚高（301笔中大量0元垃圾），用户体验差

**建议：** 过滤后的通知不应写入 transactions 表，或提供前端隐藏选项
